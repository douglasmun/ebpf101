# Chapter 18 — tc/BPF (egress, and the full sk_buff)

**Code:** [`../18-tc/`](../18-tc/)
**Build:** `cd 18-tc && make`
**Run:** `sudo ./18-tc/tcstat <interface>`

## Concept

XDP (ch17) was the *receive* datapath. **tc/BPF** is the other one, and it adds
the two things XDP can't give you:

| | XDP (ch17) | tc/BPF (ch18) |
|---|---|---|
| where | in the NIC driver, **before** an `sk_buff` exists | **after** the kernel builds the `sk_buff` |
| direction | **ingress only** | **ingress *and* egress** |
| context | `struct xdp_md` (`data`/`data_end`) | `struct __sk_buff` (rich: `len`, `protocol`, `mark`, …) |
| speed | fastest (pre-allocation) | slightly later, but more metadata |
| hosted by | the driver | the **`clsact`** qdisc |

The headline is **egress**: for the first time we can count the packets the
machine *sends*, not just what it receives. And because an `sk_buff` already
exists, `skb->len` hands us the packet length directly — byte/throughput
accounting with no header parsing.

As in ch17 this stays a pure observer: every program returns **`TC_ACT_OK`**, so
it counts and never drops or alters a packet.

## Building blocks

### Two programs, two hook points

`skb->len` per direction, summed into a per-CPU array. We attach a separate
program at each hook so the direction is unambiguous:

```c
SEC("tc")
int tc_ingress(struct __sk_buff *skb) { return account(skb, DIR_INGRESS); }
SEC("tc")
int tc_egress (struct __sk_buff *skb) { return account(skb, DIR_EGRESS);  }
```

```c
rec->packets += 1;
rec->bytes   += skb->len;   /* the byte counter XDP made you compute by hand */
return TC_ACT_OK;
```

### Attaching via the clsact qdisc

tc/BPF lives on a special qdisc called **`clsact`**, which provides both an
ingress and an egress hook. libbpf's TC API manages it:

```c
LIBBPF_OPTS(bpf_tc_hook, hook, .ifindex = ifindex,
            .attach_point = BPF_TC_INGRESS | BPF_TC_EGRESS);
bpf_tc_hook_create(&hook);                 /* install the clsact qdisc */

LIBBPF_OPTS(bpf_tc_opts, opts, .prog_fd = bpf_program__fd(skel->progs.tc_ingress));
hook.attach_point = BPF_TC_INGRESS;
bpf_tc_attach(&hook, &opts);               /* attach to the ingress hook */
/* … and again with BPF_TC_EGRESS for the egress program … */
```

`bpf_tc_hook_create` returning `-EEXIST` just means a `clsact` qdisc was already
present — harmless. On exit `bpf_tc_hook_destroy` removes the qdisc and both
filters in one call. (You can see the qdisc meanwhile with `tc qdisc show dev
<iface>` and the filters with `tc filter show dev <iface> egress`.)

## What the output reveals

Real capture on this machine, wifi (`wlp3s0`), a short browsing window:

```
DIR              PACKETS            BYTES
ingress              595           654240
egress                36             2837
```

The **`egress` row** is the new information — packets and bytes the host *sent*,
which no earlier chapter could observe. And it shows the classic shape of
receiving data with textbook clarity:

- **~639 KB in vs ~2.8 KB out** — a ~230:1 byte ratio. You pull down a page or
  stream and reply with almost nothing.
- **595 packets in vs 36 out** — the 36 egress packets are mostly cumulative
  TCP **ACKs** (one ACK acknowledges many received segments) plus a request or
  two. That asymmetry *is* how a download looks on the wire.
- Run a big **upload** instead (`scp` a file out, post a large form) and it
  flips — egress bytes pull ahead.

Because `skb->len` is the full frame length, the byte totals are real
throughput, not estimates — the kind of per-interface, per-direction accounting
you'd build a bandwidth monitor on.

## The wall (→ next)

XDP and tc both *can* act — `TC_ACT_SHOT` drops, `bpf_redirect()` forwards — but
we deliberately only observed. Two frontiers remain on the advanced track:
**tail calls** (one BPF program calling into another, to build large pipelines
and scale past the 1-million-instruction limit), and **LSM BPF** (attaching to
Linux Security Module hooks to make allow/deny decisions on kernel operations) —
the step from watching and shaping packets to enforcing policy.
