# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build system

Each chapter is self-contained. For libbpf chapters (07+), build from the chapter directory:

```bash
cd 07-libbpf && make          # build
cd 07-libbpf && make clean    # remove all generated artifacts
```

The Makefile follows the same 4-step pattern across all libbpf chapters:
1. `vmlinux.h` — generated from the running kernel's BTF via `bpftool btf dump file /sys/kernel/btf/vmlinux format c`
2. `*.bpf.o` — BPF ELF object compiled by `clang -target bpf`
3. `*.skel.h` — typed C skeleton generated by `bpftool gen skeleton`
4. userspace binary — compiled by `gcc`, linked against `-lbpf -lelf -lz`

BCC chapters (01–06) need no build step; run directly with `sudo python3 <chapter>/<file>.py`.

## Running

All programs require root:

```bash
sudo python3 01-hello-world/hello.py         # BCC chapters
sudo ./07-libbpf/execsnoop                   # libbpf chapters
```

Chapter-specific tooling (e.g. `trigger-attacks.sh`, `verify-execguard.sh`) lives in the chapter directory.

## Architecture

Two stacks, a clear boundary at ch07:

**Ch01–06 (BCC/Python):** The eBPF C is an inline string compiled at load time by BCC using the system LLVM. Kernel types come from runtime BTF. No separate build step.

**Ch07–23 (libbpf/C, CO-RE):** The eBPF C is a real `.bpf.c` file compiled at build time to a portable ELF object. `vmlinux.h` encodes every kernel type; libbpf patches field offsets at load time against the running kernel's BTF — the binary works on any BTF-enabled kernel with compatible types and helpers, without recompilation. The generated `*.skel.h` provides typed `open/load/attach/destroy` wrappers, eliminating stringly-typed libbpf calls.

**Program type progression:**
- Ch07–13: tracepoints and kprobes (process and TCP lifecycle observation)
- Ch14: verifier boundary programs (intentional load failures + fixes)
- Ch15: first uprobe (uretprobe on bash readline)
- Ch16: map pinning to `/sys/fs/bpf`, operated via `bpftool`
- Ch17: XDP (ingress datapath, `XDP_PASS`/`XDP_DROP`, per-CPU array, direct packet access)
- Ch18: tc/BPF (`clsact` qdisc, ingress+egress, `struct __sk_buff`)
- Ch19: tail calls (`PROG_ARRAY`, `bpf_tail_call`)
- Ch20: LSM BPF (`bprm_check_security`, requires `lsm=bpf` in kernel boot params)
- Ch21: XDP firewall (parse to L4, runtime-configurable blocklist map, `XDP_DROP`)
- Ch22: BPF iterator (`bpf_seq_printf`, `attach_iter` + `bpf_iter_create` + `read()`)
- Ch23: socket filter IDS (`BPF_PROG_TYPE_SOCKET_FILTER`, `SO_ATTACH_BPF` on AF_PACKET; no direct packet access — uses `bpf_skb_load_bytes`; rule engine fully in userspace)

**Key distinction — direct packet access:** XDP (ch17/21) and tc (ch18) allow `skb->data`/`data_end` with verifier bounds checks. Socket filters (ch23) do not; they must use `bpf_skb_load_bytes()`. This is enforced by the verifier.

## Environment

- Kernel 6.8, BTF at `/sys/kernel/btf/vmlinux` (CO-RE capable)
- clang 18, gcc, libbpf-dev, libelf-dev, bpftool, bpftrace
- Ch20 (LSM) requires `bpf` in the kernel `lsm=` boot parameter; `20-lsm/enable-bpf-lsm.sh` configures this (requires reboot)
